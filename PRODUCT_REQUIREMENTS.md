# 🗂️ 專案產品規格書 (PRODUCT_REQUIREMENTS.md)

本文件為「濟州島雙人旅遊行程規劃器」的最高核心規格。所有功能邏輯、UI 風格與互動細節皆以此為唯一基準。

## 🎨 1. UI 視覺風格與視覺規範
- **核心色調**：導入 `shadcn/ui` 組件庫。全站以「沙灘微米白」與「湛藍海洋色」為主體（佔 80%），營造渡假感。
- **點綴色彩**：選用「夕陽橘」僅作為 5% 的細節線條點綴（如：Hover 狀態、選中標籤、特殊高亮），絕對禁止使用在大面積背景，避免視覺疲勞。

## 📋 2. 行程卡片與景點內容規格
- **行程封面圖**：大行程卡片採用 Unsplash API 依關鍵字動態自動配圖，並將圖片網址定值存入資料庫 `cover_url` 欄位，確保每次載入圖源穩定不變。
- **內層景點圖標**：景點保留精簡、語意化的可愛 Emoji 圖標（如：🍴 餐廳, ☕️ 咖啡廳）。
- **中韓多國語言**：景點內容必須包含「中文名稱」、「韓文名稱」與「韓文地址」。
- **一鍵複製機制**：在韓文名稱與韓文地址旁，皆必須配置專屬的「一鍵複製」小按鈕。
    - **UX 反饋**：點擊複製成功後，系統必須跳出 `shadcn/ui` 的 Toast 提示（內容範例：`「韓文地址已複製，快去 Naver Map 貼上吧！🍊」`）。
- **必吃與遊記雙軌制 (A/B方案)**：
    - **A方案**：支援點擊展開看「親筆備忘 / 必吃小筆記」。
    - **B方案**：支援填入外部食記/部落格網址，系統需自動擷取 OGP（Open Graph Protocol）產生網頁預覽縮圖。

## 🔐 3. 公開行程分享、暗號解鎖與 RLS 防禦
- **【SSR 密碼攔截】**：實作免登入分享，透過 Next.js 伺服器端渲染 (SSR) 保護。當訪客進入「暗號模式」網址時，若瀏覽器無正確 Cookie，Server 端必須徹底攔截，僅渲染「專屬密碼輸入框（提示：請輸入生日/紀念日等小暗號）」，絕對禁止將行程資料傳送到前端，防範透過瀏覽器開發者工具（DevTools）偷看後端隱私資料。
- **【Cookie 隔離防禦機制】**：驗證成功後寫入的 Cookie 鍵名**必須動態綁定行程 ID**（型態如：`trip_unlock_[id]`）。嚴禁全站共用單一 Token，確保解鎖行程 A 的瀏覽器，在未輸入正確暗號前，無法直接觀看私密行程 B。
- **【Cookie 分流與路人模式】**：網址後方若帶有 `?mode=public` 則為「純路人唯讀參觀模式」，系統自動分流繞過密碼攔截。密碼輸入頁面下方必須提供「切換為路人參觀模式」的退路按鈕，防止外人誤觸時卡死。

## 🙅‍♂️ 4. 景點表情微互動與防呆互斥規則
本模組定義了使用者（解鎖者/小傑）在互動模式下，對各景點卡片點擊表情符號時的控制邏輯。**在路人模式下，所有表情數值與以下狀態皆為「唯讀呈現」，按鈕鎖死（Disabled）無法點擊，且不顯示清除叉叉。**

### A. 表情符號配置與行為
- 😍 **【超想去！】**：權重累積型。支援無限次重複點擊累加（+1, +2, +3...）。
- 🫩 **【還好ㄟ】**（單片眼鏡評估臉）：權重累積型。支援無限次重複點擊累加（+1, +2, +3...）。
- 🛒 **【買！】**：權重累積型。支援無限次重複點擊累加（+1, +2, +3...）。
- 🙅‍♂️ **【小傑覺得先不要】**：生死開關型。嚴格實作 Toggle 機制（數值僅在 0 與 1 之間切換）。

### B. 狀態清空與互斥邏輯 (Mutual Exclusion Rules)
- **【😍 超想去！】 與 【🛒 買！】**：🟢 可並存。
- **【🫩 還好ㄟ】 與 【🛒 買！】**：❌ 互斥。點擊 🫩 時，🛒 的計數強制清空歸零。
- **【😍 超想去！】 與 【🫩 還好ㄟ】**：❌ 互斥。點擊 😍 則 🫩 歸零；點擊 🫩 則 😍 歸零。
- **【🙅‍♂️ 先不要】 與 其他正面表情**：❌ 絕對互斥。
    - 當點擊 🙅‍♂️ 時，不論 😍, 🫩, 🛒 原本累積到多少次，系統必須立刻將這三項數值**全數清空歸零**。同時，該景點卡片即時觸發透明度降低（`opacity-50`）與刪除線（`line-through`）視覺效果，並浮現「🙅‍♂️ 小傑覺得先不要」專屬提示小字。
    - 反之，若卡片處於 🙅‍♂️ 鎖定狀態，使用者一旦點擊 😍, 🫩, 🛒 任何一者，卡片立刻恢復亮色正常外觀，🙅‍♂️ 歸零，被點擊的新表情起始值設為 1。

### C. 多點擊取消機制 (Reset Mechanism)
- **顯示條件**：當權重累積型表情（😍, 🫩, 🛒）的累加數字大於或等於 1，**且當前瀏覽者具備點擊權限（非路人模式）時**，該按鈕右上角才會動態渲染出一個極小的灰色清除叉叉按鈕。
- **執行行為**：點擊該清除叉叉時，該單一表情項目的計數器會直接「清空歸零」，且不影響其他處於並存狀態的表情。

## 🆕 5. 新增行程（人工新增 / AI 生成）與 Unsplash 封面自動配圖
於 `/trips`（個人行程總覽大廳）提供一顆「**新增行程**」按鈕，點擊後**先讓使用者選擇本次要用哪種方式建立**：**✍️ 人工新增** 或 **🤖 AI 生成**。

### A. 入口、模式選擇與可見性
- 入口：`/trips` 大廳的「**新增行程**」按鈕 → 彈出模式選擇（人工新增 / AI 生成）。
- **可見性**：此按鈕位於**受保護的 `/trips`**，僅**登入者可見**；訪客（未登入）依 ADR-0006 一進 `/trips` 即被導回首頁，故看不到此按鈕。
- **多使用者隔離**：不分帳號，**任何登入者**皆可新增行程；行程以 `user_id` 綁當前登入者，透過 RLS 天生隔離（A 看不到 B 的私密行程）。未來開放註冊後，此機制**無需改動即可支援多人**。

### B. 模式一：✍️ 人工新增
- 開啟一般表單（Modal）：行程標題、描述、公開/私密等基本欄位。
- **封面圖（人工串接 Unsplash，關鍵字可跨語言）**：
    - 使用者可輸入**中／韓／英**任一語言關鍵字（例如 `涯月邑`、`애월읍`、`jeju cafe`）。點「生成封面」→ 後端先用 **Gemini** 將關鍵字正規化為英文（羅馬拼音，例如 `涯月邑` → `Aewol-eup`；非英文才翻譯，本身是英文則略過）→ 再以英文關鍵字呼叫 **Unsplash API** 取一張高畫質風景圖並預覽。
    - Gemini 正規化時可一併回傳 `zh／ko／en` 三種寫法；其中 `ko`（韓文）未來可協助填入景點的韓文名稱欄位（呼應第 2 節中韓雙語）。
    - 若 Unsplash 對過細的地名圖片太少，後端可退用較廣的關鍵字（如 `Aewol Jeju`、`Jeju cafe`）作為備援。
    - 支援「**重新生成**」多次換圖，直到滿意；選定後將該圖 URL 存入 `cover_url`。封面亦可**留空**（不強制）。
    - **額度防呆**：每次「生成／重新生成」都算一次 Unsplash 呼叫（若觸發翻譯亦各算一次 Gemini 呼叫）；Unsplash 免費 Demo 額度約 **50 次/小時**（以官方後台實際顯示為準）。生成中按鈕須 `disabled`；達限或失敗時顯示友善提示（例如「本小時封面圖生成額度已用完，請稍後再試」），避免連續狂點爆額度。
- 送出後於 Supabase 建立行程（`destinations` 由使用者稍後於編輯器自行新增）。
- 成功後導向 `/trips/[id]` 進行後續編輯。
- 此路徑**不使用 AI 生成**，但封面圖會用到 Unsplash；核心欄位（標題／描述）本身不需任何外部金鑰即可建立。

### C. 模式二：🤖 AI 生成
- 開啟願望輸入 Modal：使用者填寫自然語言願望，例如：「我想去濟州島 4 天，想去涯月邑喝咖啡，排懶人自駕行程」。
- **Loading 防呆**：送出後按鈕即 `disabled` 並顯示 Loading 狀態，避免重複送出；生成失敗需顯示錯誤訊息。
- 後端生成流程（金鑰一律只在伺服器端）：
    1. Modal 送出 → 呼叫 Next.js Route Handler `POST /api/trips/generate`。
    2. 後端以 **Google Gemini API**（Google AI Studio 免費額度）生成，要求**嚴格回傳結構化 JSON**：行程標題、天數、每日景點列表（中文名稱等）、以及一個**代表該行程的英文關鍵字**。
    3. 後端拿該英文關鍵字向 **Unsplash API** 取一張高畫質風景圖 URL。
    4. 透過 Supabase **伺服器端**實例寫入 `trips`（含 `cover_url` 定值）與 `destinations`（AI 生成的景點）。
    5. 成功後將使用者導向新行程頁面 `/trips/[id]`。

### D. AI 產出僅為「初稿」，使用者保有完全編輯權（重要）
- AI 生成的景點**僅為起始草稿，非最終版**。使用者在 `/trips/[id]` 編輯器中必須能對景點進行：
    - **人工新增**、**於任意位置中間插入**、**修改**、**刪除**、**重新排序**。
- 即：AI 只負責「快速起草」，最終行程內容以使用者人工編輯為準（此編輯能力屬 `/trips/[id]` 編輯器範疇，與模式一建立後的編輯共用同一套）。

### E. 安全與金鑰
- Gemini 與 Unsplash 的 API Key **一律為伺服器端環境變數（不得加 `NEXT_PUBLIC_` 前綴）**，只在 Route Handler 內使用，嚴禁外洩到前端。
- **封面取圖統一走後端**：人工新增（B）與 AI 生成（C）取 Unsplash 圖片時，皆透過伺服器端 Route Handler 呼叫，前端不得直接持有 Unsplash Key。
- **Unsplash 額度**：免費 Demo 約 50 次/小時（以官方後台為準）；人工「重新生成」與 AI 生成都各算一次呼叫，UI 需防呆並在達限時友善提示。
- 寫入 `trips` / `destinations` 時，`user_id` 綁定當前登入者，受既有 RLS 保護。
