# CLAUDE.md - Developer Collaboration Rules

## 🤝 互動人格與溝通原則
1. **嚴格放慢腳步**：禁止一次給出多個階段的程式碼。必須採取「單一細項引導 ➔ 等待使用者回報 ➔ 給出下一步程式碼」的模式。
2. **定時關心進度**：每次回覆的結尾，必須明確詢問使用者當前停留在哪一個特定操作，確認本機狀態後才前進。
3. **格式化排版**：拒絕密集的文字牆。必須善用標題、水平分割線（---）、粗體、以及程式碼區塊，維持極高的可讀性與可掃視性。
4. **誠實坦承**：網頁版 AI 助手若不具備直接讀取或透視負責人本機檔案之能力，嚴格禁止瞎掰或產生幻覺，被糾正時應立即校正回歸。

## 💻 程式碼編寫與型別規範
1. **嚴禁程式碼偷跑**：在對應的資料庫資料表（Table）尚未在 Supabase 建立前，禁止提供呼叫該資料表的 React 前端程式碼。
2. **Vue 2 轉型友善對比**：在解釋 Next.js 的 SSR、Middleware 或狀態管理時，可適度用 Vue 2 的 `mounted` 或 `router.beforeEach` 做白話對比，幫助使用者快速理解架構本質。
3. **安全性第一**：絕對禁止在程式碼範例中硬編碼（Hardcode）任何真實密碼或私鑰，必須引導使用者安全地使用 `.env.local`。
4. **環境變數對接**：提供程式碼時，必須確保變數名稱與本機環境完全一致（例如 `NEXT_PUBLIC_SUPABASE_URL`）。

## 📜 Git Commit 訊息規範 (Conventional Commits)
當自動化工具（如 Claude Code）自動生成、撰寫，或是負責人手動提交 Git Commit 訊息時，必須嚴格遵守以下格式字首，且一律使用**「繁體中文」**進行細項描述：

| 字首 (Type) | 適用時機 (When to use) | 範例 (Example) |
| :--- | :--- | :--- |
| **`feat:`** | 新增功能、實作新頁面、新增 API 連線邏輯 | `feat: 實作 /login 帳密登入頁面與狀態跳轉` |
| **`fix:`** | 修復 Bug、修正錯誤的邏輯、修正網頁報錯 | `fix: 修正 middleware 沒登入會無限重導向的 Bug` |
| **`docs:`** | 純文件修改（如更新 README, DEVELOPMENT_PLAN 等） | `docs: 更新開發進度表，調整 Google OAuth 時程` |
| **`style:`** | 不影響程式邏輯的畫面修改、CSS 排版、排版美化 | `style: 微調登入按鈕的陰影與間距，改為極簡風` |
| **`refactor:`**| 重構程式碼（不影響功能，純粹把 Code 寫得更漂亮、抽離邏輯） | `refactor: 將 Supabase 驗證邏輯抽離成獨立的 Hook` |
| **`chore:`** | 建置流程或輔助工具的變動（如安裝新套件、改 npm 設定） | `chore: 安裝 @supabase/ssr 套件` |
| **`types:`** | 純粹修改 TypeScript 的型別定義檔（`.ts` 宣告） | `types: 重新執行一鍵生成，更新最新資料庫型別` |

⚠️ **Commit 遵守守則**：
1. 字首一律小寫，冒號後面必須**空一格**（例如：`feat: ...`）。
2. 盡可能在下方使用條列式（- ）簡述本次變動了哪些檔案、資料表或業務邏輯。
